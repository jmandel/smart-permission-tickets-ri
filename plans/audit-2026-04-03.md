# Reference Implementation Audit: fhir-server vs input/ spec

**Date:** 2026-04-03

This note reviews `reference-implementation/fhir-server` against the Permission Ticket material in `input/` and records what we should actually do next for this repo.

This is a **reference implementation**, not a full product server. So the bar here is:

- be correct for the flows we claim to support
- make protocol boundaries clear
- avoid pretending to implement use cases we do not yet cover
- explicitly defer broader-spec items instead of silently ignoring them

At the moment, the server is intentionally centered on:

- UC1-style network patient access
- network RLS resolution
- site-bound token exchange
- constrained read/search over a synthetic corpus

---

## Scope Framing

### Supported Now

- signed Permission Tickets for `network-patient-access-v1`
- SMART token exchange at network and site surfaces
- sender-constrained client auth via `private_key_jwt`
- site/org/jurisdiction/date/sensitivity filtering
- network directory + `$resolve-record-locations`
- built-in demo landing page and viewer

### Not Yet a Goal

- full spec coverage across all Permission Ticket use cases
- revocation / CRL operations
- external issuer federation and dynamic JWKS fetch
- every possible FHIR resource/search parameter
- every signature algorithm the spec could allow

---

## Already Addressed Since Early Review

These are no longer open concerns for the current codebase:

### [x] A1. Public origin should be explicit, not inferred from proxy headers

The server now uses `PUBLIC_BASE_URL` as the explicit public origin for:

- SMART config URLs
- issuer metadata / JWKS URLs
- network and site FHIR base URLs
- token audiences

`X-Forwarded-*` origin reconstruction is no longer part of the default implementation path.

Why this matters:

- better security
- deterministic hosting behavior
- clearer deployment model for a reference implementation

## Correctness / Protocol Shape

These are worth taking on in the near term because they improve correctness and interoperability for the flows we already support.

### [x] D1. Validate `subject_token_type`

**Implemented**

- `/token` now requires `subject_token_type`
- the accepted value is `https://smarthealthit.org/token-type/permission-ticket`
- missing or unsupported values return OAuth-style `invalid_request` errors

### [x] D2. Return OAuth error JSON from `/token`

**Implemented**

- `/token` now returns OAuth JSON errors instead of FHIR `OperationOutcome`
- current categories include:
  - `invalid_request`
  - `invalid_grant`
  - `invalid_scope`
  - `unsupported_grant_type`
  - `invalid_client`

### [x] D3. Honor requested `scope` narrowing

**Implemented**

- requested `scope` is now intersected with the Permission Ticket scopes
- narrowing works for both network-level and site/global token exchange
- widening beyond the ticket returns `invalid_scope`

### [x] D4. Add explicit presence checks for `subject_token`

**Implemented**

- missing or empty `subject_token` is rejected explicitly as `invalid_request`
- the server uses a specific `No permission ticket provided` error path instead of falling through JWT parsing

### [x] D5. Add explicit presence checks for `ticket_type`

**Implemented**

- missing `ticket_type` is now rejected explicitly
- `Unsupported ticket type` is reserved for present-but-unknown values

### [x] D6. Document implementation-specific ticket extensions

**Implemented**

The docs now call out these local extensions explicitly:

- `details.dateSemantics`
- `details.sensitive.mode`

They are treated as implementation-defined behavior in the reference server, not as generic base-spec fields.

## Broader Read/Search Coverage

These are not protocol-blocking, but they matter for the usefulness of the reference implementation and the viewer.

### [x] N1. Broaden search coverage for supported resource types

**Implemented**

The main search matrix now includes deliberate search coverage for:

- `AllergyIntolerance`
- `Procedure`
- `Immunization`
- `ServiceRequest`

This is covered by integration tests using representative patient-facing searches.

### [ ] N2. Consider `Task` later, only if a supported use case needs it

`Task` should not be added just for completeness. Add it when we take on a use case where it is truly part of the demonstrated access flow.

## Explicitly Deferred

These are valid ideas from the spec/audit, but they are not necessary yet for a clear and correct reference implementation of the currently supported flow.

### [ ] X1. Revocation / CRL checking

This is real protocol work, but it adds:

- remote fetch behavior
- caching policy
- fail-closed network semantics
- more moving parts in a local demo

We should acknowledge it, but defer it until we want to model revocable tickets explicitly.

### [ ] X2. External issuer federation and dynamic JWKS fetch

Right now the issuer is intentionally in-stack and explicitly configured.

For this repo, that is acceptable because:

- we are demonstrating the issuer role
- we are not yet trying to demonstrate federated trust relationships

We should revisit this when we move beyond the built-in issuer model.

### [ ] X3. RS256 support

The spec may allow more than ES256, but this reference implementation should not broaden signature support until it has a concrete reason to do so.

Current stance:

- ES256 only
- clearly documented
- acceptable for current scope

### [ ] X4. Per-ticket-type `cnf` requirement matrix beyond current supported flow

There is a real rule matrix here, but for now:

- we support one main ticket type
- the demo issuer issues bound tickets
- strict/key-bound semantics are already demonstrated

This becomes more important as soon as we add more ticket types.

### [ ] X5. Strict validation of prohibited subject fields

This is a conformance nicety, not a high-value correctness fix right now. The subject resolver already only reads the fields relevant to the selected subject type.

### [ ] X6. Advertising `client_credentials`

Do not advertise flows we do not actually want to demonstrate.

If the server is not using `client_credentials` as part of its intended SMART surface, it should not be added just because an example elsewhere includes it.

## What the Server Already Does Well

- SMART-style token exchange at network and site surfaces
- ES256 ticket signing + verification
- sender-constrained key binding with `cnf.jkt`
- all three subject resolution patterns used by the demo:
  - `match`
  - `identifier`
  - `reference`
- correct intersection behavior across:
  - subject
  - sites
  - organizations
  - jurisdictions
  - dates
  - sensitivity
- site-bound token issuance and audience enforcement
- network RLS resolution that returns only visible sites
- clear separation between:
  - issuer role
  - network directory / RLS role
  - site clinical data role
- dynamic client registration with `private_key_jwt`
- built-in demo + viewer that exercise the protocol flow end to end

## Recommended Priority Order

### Priority 1: Fix protocol-shape issues for supported flows

1. [x] OAuth-style `/token` errors
2. [x] `subject_token_type` validation
3. [x] requested `scope` intersection
4. [x] explicit `subject_token` presence check
5. [x] explicit `ticket_type` presence check

### Priority 2: Improve reference-implementation usefulness

6. [x] broaden search coverage for supported resource types
7. [x] document local ticket extensions clearly

### Priority 3: Future expansion, explicitly deferred for now

8. [ ] revocation / CRL
9. [ ] external issuer federation / JWKS fetch
10. [ ] RS256
11. [ ] broader ticket-type matrix enforcement
12. [ ] `Task` and additional use-case-specific resource support

---

## Bottom Line

The main correctness items for the currently supported flow are now addressed. The remaining work is mostly deliberate deferral:

- do not expand beyond the supported use cases just for completeness
- revisit revocation, external trust, and broader signature algorithms when the demo scope truly needs them

That remains the right balance for this repo right now.
